# Apartment Metrics PoC

Proof of Concept (PoC): агентная симуляция повседневных перемещений по планировкам квартир для расчёта метрик «комфорта». Три профиля агентов перемещаются по реальным планам, представленным как графы комнат; для каждого агента за один день считаются четыре метрики.

---

## Графовая модель планировок

Каждая квартира — это неориентированный взвешенный граф, собранный из `full_json/<id>.json`:

- **Узлы (nodes)** — комнаты (с именами и координатами центроидов в пикселях).
- **Рёбра (edges)** — связи между комнатами с двумя весами: `distance` (пиксели между центроидами) и `door_centroid` (опорная точка двери для расчёта углов, если проем есть).
- **Маршрутизация** — кратчайший путь через Дейкстру из NetworkX (`weight="distance"`).
- **Сопоставление комнат (room mapping)** — канонические типы (`спальня`, `кухня`, `санузел`, …) маппятся на реальные названия комнат в конкретной планировке через `mapping_rules` во встроенном `rooms_cfg`. Если в планировке нет явной спальни, используется первый доступный алиас (`комната`, `комната 1`, …) как fallback.

Используются 5 реальных планировок CubiCasa5K (ID: 30, 413, 476, 1042, 1732), от 4 до 9 комнат.

---

## Агенты

Три профиля. У каждого есть дневной сценарий (упорядоченный список перемещений) и коэффициенты чувствительности `k_dist`, `k_trans`, `k_angle`, которые масштабируют соответствующие компоненты Friction Score.

| Агент | Дневной сценарий | k_dist | k_trans | k_angle |
|---|---|---|---|---|
| **elderly** | спальня→кухня (×3), кухня→СУ (×3), спальня→прихожая, гостиная→спальня (×3) | 1.5 | 1.5 | 1.1 |
| **young_couple** (2 агента) | спальня→СУ (×2), кухня→гостиная (×3), гостиная→спальня | 1.0 | 1.0 | 1.0 |
| **bachelor** | спальня→рабочее место, рабочее место→кухня (×6), рабочее место→СУ (×6), кухня→гостиная (×2) | 1.0 | 1.0 | 1.0 |

Одно перемещение — это один проход по кратчайшему пути между двумя комнатами. Так как граф неориентированный и веса симметричны, «туда‑обратно» отдельно не моделируется: дистанция/переходы будут теми же. Параметр `repeat` используется только как «сколько раз за день случается такой проход», без удвоения из-за направления.

Для `young_couple` запускаются два экземпляра с одинаковым сценарием. У каждой команты проставлен признак невозможности одновременного использования комнаты (СУ, работчее пространство): если один агент занимает комнату в слоте, второй получает штраф `overlap`.

## Metrics

**Daily Distance** — сумма длин кратчайших путей по всем перемещениям за день. Отражает общий объём ходьбы.

**Number of Transitions** — общее число «переходов между комнатами» (сколько рёбер графа пройдено) по всем перемещениям. Отражает количество пересечений границ комнат независимо от дистанции.

**Average Path Length** — средняя длина одного маршрута. Малое значение означает компактную планировку, где комнаты расположены ближе друг к другу.

**Friction Score** — нормированный агрегированный индекс в диапазоне [0, 1], равновзвешенная сумма четырёх компонентов:

| Компонент | Что отражает | Нормализация |
|---|---|---|
| `hd = D / D_ref` | Нагрузка по дистанции | `D_ref` = максимум дневной дистанции по всем планировкам |
| `ht = T / H_ref` | Нагрузка по переходам | `H_ref` = максимум переходов по всем планировкам |
| `ha = A / A_ref` | Угловая нагрузка на дверях. Сумма синусов углов. | `A_ref` = максиальное кол-во переходов в наборе с уголом 90 или sin=1 |
| `overlap` | 1, если два агента конкурируют за комнату недоступную к одновременному использованию в одном временном слоте | бинарный |

Итоговая формула:

`Friction = 0.25 · (k_dist·hd + k_trans·ht + k_angle·ha + k_overlap·overlap)`

Смыслы коэффициентов:

- **`k_dist`**: чувствительность к дистанции (при `hd`)
- **`k_trans`**: чувствительность к числу переходов (при `ht`)
- **`k_angle`**: чувствительность к поворотам/углам у дверей (при `ha`)
- **`k_overlap`**: штраф за конкуренцию за эксклюзивные комнаты (при `overlap`)

Коэффициенты `k_c ≥ 1` усиливают вклад конкретного фактора.

## Итоги

Результаты симуляции (`df_agg`) для 5 планировок × 3 наборов агентов (bundles):

### Рейтинг планировок по сценарию

**Bachelor (холостяк, работает из дома)**
Высокая ежедневная дистанция из-за насыщенного сценария (кухня ↔ рабочая зона ↔ санузел × 6).
Лучшие планировки — с компактным расположением ключевых комнат и короткими путями между ними.
Оценка напрямую коррелирует с `Average Path Length`: планировки с более короткими средними путями всегда выигрывают.

**Elderly (пожилой, минимальный сценарий)**
Самый «тихий» агент. Различие между планировками выражено слабее, чем у других сценариев.
Принципиально важна лишь доступность спальни и санузла без длинных обходов.

**Young couple (молодая пара, два агента)**
Единственный сценарий с возможным штрафом за наложение (overlap).
Планировки с единственным санузлом получают `overlap = 0.5`, что сразу даёт прибавку к Friction Score.
Для этого сценария критична наличие второго санузла или большой площади для развода маршрутов.

---

### Самая универсальная планировка
Лучшей по среднему Friction Score оказывается **plan 476** (4 комнаты).
Несмотря на минимальное число комнат, планировка компактна — все ключевые узлы близко.
Короткий средний путь снижает все три нормированных компонента (hd, ht, ha).
Для молодой пары присутствует overlap, однако он перекрывается низкими значениями остальных компонент.

### Самая неудачная планировка
**Plan 30** (6 комнат) показывает наибольший Friction Score у большинства агентов.
Причина — высокая `Average Path Length`: комнаты расположены далеко друг от друга.
Для молодой пары дополнительно возникает overlap из-за единственного санузла.
Число комнат (6) не даёт преимуществ — хуже, чем планировки с 9 комнатами (1042, 413), где маршруты оказались короче.

---

### Общий вывод
Метрика Friction Score и её компоненты показывают, что **компактность планировки важнее числа комнат**.
Количество комнат само по себе не определяет удобство — ключевую роль играет длина пути между функциональными зонами.
Планировки с несколькими санузлами однозначно предпочтительны для совместного проживания.

## Ограничения

- Дистанции — это условное измерение из SVG-координат CubiCasa5K. Относительные сравнения между планировками валидны; абсолютные значения — нет.
- Числа повторов в сценариях (`repeat`) — вручную заданные константы, не полученные из реальных данных использования.
- Угловая нагрузка (`ha`) считается только для рёбер, где в JSON есть `door_centroid`. Планировки без этого поля дают `ha = 0`, занижая штраф за повороты.
- `D_ref` и `H_ref` вычисляются по выбранным 5 планировкам. Добавление новых планов пересчитает шкалу (и, соответственно, все Friction Score).
- Симулируется один день.
